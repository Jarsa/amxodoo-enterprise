# CLAUDE.md — l10n_mx_edi_cfdi_payment_followup

## Overview

Odoo 17.0 module that automates tracking, validation, and operational control of CFDI
Payment Complements (Complementos de Pago PPD) for:

- **Vendor payments** — request, receive, validate, and track supplier-issued payment
  complements.
- **Customer collections** — detect payments requiring a complement and trigger its
  generation.

The module eliminates manual work, reduces fiscal errors, and provides a single status
field to monitor every outstanding complement across the organization.

## Functional Scope

### What this module does

1. Adds a **CFDI payment state** (`l10n_mx_edi_cfdi_payment_state`) to `account.move`,
   automatically calculated on reconciled payments.
2. Provides a **manual button** on `account.payment` and `account.bank.statement.line`
   to send a complement request email to the configured vendor contact.
3. Processes any **XML attachment** added to a payment record: parses it with `lxml`,
   validates it against Odoo data, and transitions the state accordingly.
4. Creates an **internal activity** when validation fails, assigned to the responsible
   user configured in accounting settings.
5. Exposes a **backend method** (`action_cfdi_payment_followup`) ready to be called by a
   scheduled action (cron) for automatic follow-up reminders.
6. Provides a **multi-company start date** setting to avoid reprocessing historical
   records.

### What this module does NOT do (out of scope)

- SAT XML automatic download
- STP/FTP banking integration
- Automatic SAT vs bank reconciliation
- OCR on PDF documents

## Technical Architecture

### Base model

All logic lives on **`account.move`**.

`account.payment` and `account.bank.statement.line` both use `_inherits` from
`account.move`, so adding fields and methods to `account.move` makes them immediately
available on both sub-models without duplication.

The views for the button and status column are added via `inherit` on:

- `account.payment` form and list views
- `account.bank.statement.line` list view (used in bank reconciliation)

### Python XML library

Use **`lxml.etree`** (available in Odoo's environment) for all CFDI XML parsing. Do not
add `satcfdi` or `cfdiclient` as dependencies.

## Models

### `account.move` (`models/account_move.py`)

#### New fields

```python
l10n_mx_edi_cfdi_payment_state = fields.Selection(
    selection=[
        ("not_required", "Not Required"),
        ("pending", "Pending"),
        ("requested", "Requested"),
        ("received", "Received"),
        ("validated", "Validated"),
        ("error", "Error"),
    ],
    string="CFDI Payment State",
    compute="_compute_l10n_mx_edi_cfdi_payment_state",
    store=True,
    readonly=False,   # allow direct writes from XML processor
    copy=False,
    tracking=True,
)

l10n_mx_edi_cfdi_payment_last_followup_date = fields.Datetime(
    string="Last Follow-up Date",
    copy=False,
    readonly=True,
)
```

#### Existing field reused

`l10n_mx_edi_cfdi_uuid` (already defined in `l10n_mx_edi`) stores the UUID of the
validated complement. Do not redefine it; reference it directly.

#### Computed state logic (`_compute_l10n_mx_edi_cfdi_payment_state`)

Trigger depends on: `payment_state`, `move_type`, `l10n_mx_edi_payment_policy`,
`line_ids.matched_debit_ids`, `line_ids.matched_credit_ids`, `l10n_mx_edi_cfdi_uuid`,
`attachment_ids`.

Rules evaluated in order:

| Condition                                                    | Resulting state             |
| ------------------------------------------------------------ | --------------------------- |
| Record is not a payment-related journal entry                | `not_required`              |
| Record date < company start date                             | `not_required`              |
| No related invoice has `l10n_mx_edi_payment_policy == 'PPD'` | `not_required`              |
| `l10n_mx_edi_cfdi_uuid` is set                               | `validated`                 |
| State already set to `error` or `requested`                  | preserve (do not overwrite) |
| None of the above                                            | `pending`                   |

> `store=True` + `readonly=False` allows the XML processor to write state directly. The
> compute method must guard with
> `if record.l10n_mx_edi_cfdi_payment_state in ('error', 'requested')` to avoid
> overwriting manually-set states.

**Un-reconcile rule:** Si un pago se des-concilia de una factura PPD, si todavía no
tiene complemento no pasa nada. Pero si ya tiene un complemento adjunto validado
(`l10n_mx_edi_cfdi_uuid` asignado), el sistema debe notificar en el chatter del pago
indicando que se ha roto la conciliación pero ya existía un complemento fiscal asociado.

#### Method: `action_request_cfdi_complement(self)`

Visible as button "Request Complement" on vendor payments and bank statement lines.
Active only when `l10n_mx_edi_cfdi_payment_state in ('pending', 'error')`.

Actions:

1. Find the contact with partner category tag `cfdi_complement_contact` on the vendor,
   or fall back to the partner's main email.
2. Send mail template `mail_template_cfdi_complement_request`.
3. Attach the first PDF attachment on the record (bank payment proof), if any.
4. Attach a plain-text list of related invoice references and amounts.
5. Set `l10n_mx_edi_cfdi_payment_state = 'requested'`.
6. Set `l10n_mx_edi_cfdi_payment_last_followup_date = fields.Datetime.now()`.
7. Post a chatter message with the email summary.

#### Method: `action_cfdi_payment_followup(self)`

Called by scheduled action (cron) or manually from a button.

Loops over records where:

- `l10n_mx_edi_cfdi_payment_state == 'requested'`
- `l10n_mx_edi_cfdi_payment_last_followup_date` older than the follow-up interval
  (default 5 days, stored in `ir.config_parameter` key
  `l10n_mx_edi_cfdi_payment_followup_interval_days`).
- Record date >= company start date.

For each matching record: calls `action_request_cfdi_complement()`.

#### Method: `_process_cfdi_payment_xml(self, attachment)`

Triggered automatically **exclusively** when an XML file is attached to the record via
`_message_post_after_hook` (do not override `write` on `attachment_ids` to avoid double
execution).

Process:

1. Check that the attachment name ends in `.xml` or `mimetype` is `application/xml`.
   Skip silently otherwise.
2. Parse with `lxml.etree.fromstring(attachment.raw)`.
3. Validate `TipoDeComprobante == "P"`. Skip silently if not.
4. Extract UUID from `tfd:TimbreFiscalDigital/@UUID`.
5. Extract `pago20:Pago` node(s).
6. Run all validation rules (see below). Collect all error messages.
7. **On success** (no errors):
   - Write `l10n_mx_edi_cfdi_uuid = uuid`.
   - Write `l10n_mx_edi_cfdi_payment_state = 'validated'`.
   - Post chatter success message.
8. **On failure**:
   - Write `l10n_mx_edi_cfdi_payment_state = 'error'`.
   - Call `_create_cfdi_payment_activity(combined_error_message)`.
   - Post chatter error message with all validation failures listed.

#### Method: `_create_cfdi_payment_activity(self, error_message)`

Creates a `mail.activity` of type `mail.activity_data_todo` with:

- Summary: `"CFDI Complement Error"`
- Note: `error_message`
- Assigned user: `l10n_mx_edi_cfdi_responsible_user_id` from `res.config.settings` for
  current company. (This setting is mandatory, so there is no fallback to `env.user`).

## XML Validation Rules

All rules run inside `_process_cfdi_payment_xml`. Collect all failures in a list and
report them together (do not stop at first error).

Module-level namespace constant:

```python
_CFDI_NSMAP = {
    "cfdi": "http://www.sat.gob.mx/cfd/4",
    "pago20": "http://www.sat.gob.mx/Pagos20",
    "tfd": "http://www.sat.gob.mx/TimbreFiscalDigital",
}
```

Always use `tree.find(".//tfd:TimbreFiscalDigital", _CFDI_NSMAP)` — never build XPath
strings by concatenation.

| #   | What to check                                                                                                  | Error message                                                    |
| --- | -------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| 1   | `TipoDeComprobante == "P"`                                                                                     | Skip silently                                                    |
| 2   | UUID present in `TimbreFiscalDigital/@UUID`                                                                    | `"UUID not found in XML"`                                        |
| 3   | At least one `pago20:Pago` node                                                                                | `"No Pago node found"`                                           |
| 4   | `pago20:Pago/@FormaDePagoP == "03"` (SPEI)                                                                     | `"Payment form is not SPEI (03), found: {value}"`                |
| 5   | `pago20:Pago/@FechaPago` (date part) >= Odoo record `date` (tolerance: configurable days forward, 0 days back) | `"Complement date {fecha} precedes payment date {date}"`         |
| 6   | Each `DoctoRelacionado/@IdDocumento` UUID matches a reconciled invoice `l10n_mx_edi_cfdi_uuid`                 | `"Invoice UUID {uuid} not found in reconciled invoices"`         |
| 7   | `DoctoRelacionado/@ImporteSaldoInsoluto == "0.00"`                                                             | `"Outstanding balance not zero for invoice {uuid}: {value}"`     |
| 8   | Sum of all `pago20:Pago/@Monto` matches Odoo payment amount (tolerance: +/- 1.00)                              | `"Amount mismatch: XML total {xml_total} vs Odoo {odoo_amount}"` |

**Idempotency & Replacements:**

- If `l10n_mx_edi_cfdi_uuid` already equals the UUID in the incoming XML, skip
  processing entirely and return without error or state change (idempotent).
- If `l10n_mx_edi_cfdi_uuid` is already set but the incoming XML has a **different
  UUID** (a replacement), verify if the previous UUID was canceled. In any case, accept
  the new UUID, but notify the responsible user (via Activity or Chatter) that a
  replacement complement was received and two UUIDs exist for the same payment.

**Date tolerance** is read from `ir.config_parameter` key
`l10n_mx_edi_cfdi_payment_date_tolerance_days` (integer, default `2`). Parse `FechaPago`
to `date` (extract `.date()` from ISO 8601 datetime) before comparing.

### `tests/common.py`

Base class `TestCfdiPaymentFollowupCommon(SavepointCase)`:

1. Creates a Mexican company with `MXN` currency.
2. Creates a vendor `res.partner` with email address.
3. Creates a product.
4. Creates a PPD vendor bill (`account.move`, `move_type='in_invoice'`) in `posted`
   state with `l10n_mx_edi_payment_policy='PPD'` and a valid `l10n_mx_edi_cfdi_uuid`.
5. Creates a bank journal and account.
6. Creates a bank statement line reconciled against the vendor bill.
7. Provides helper
   `_build_cfdi_xml(uuid, payment_date, amount, invoice_uuids, forma_pago="03", saldo_insoluto="0.00")`
   that returns a minimal but structurally valid CFDI 4.0 Complemento de Pago (Pagos
   2.0) as `bytes` using `lxml.etree.tostring`.
8. Provides helper `_attach_xml(record, xml_bytes, filename="complement.xml")` that
   creates an `ir.attachment` linked to `record` and calls
   `record._process_cfdi_payment_xml(attachment)` directly for unit testing without
   triggering the mail hooks.
